ResourceWorldResetter
Documentation

A Spigot plugin for automated resource-world resets. Features phase-aware state tracking, a unified /rwr command tree, safe player teleportation, and region-level selective resets — all without server restarts.

🟢 Java 25+ required
🟣 Spigot API 26.1+
🔵 Multiverse-Core 4.3.1+
⚡ BSD-3-Clause License

📖Introduction

ResourceWorldResetter automates the deletion and regeneration of a designated resource world on your Minecraft server. Players can mine, build, and exhaust resources freely, knowing the world will reset automatically on a defined schedule — daily, weekly, or monthly — without requiring a full server restart.

Version 4.0.0 is a major breaking change release that introduces a unified /rwr command structure, an explicit phase-based reset state machine, versioned configuration, and improved operator visibility into reset progress and failures.

⚠️
Breaking Change Release

All legacy commands (/rwrgui, /resetworld, /rwrregion, /rwrresume, /reloadrwr) are no longer registered. You must update all scripts, cron jobs, and automation to use the new /rwr subcommand tree.

What's New in v4.0.0

🔁

Phase-Based State Machine

Resets progress through explicit phases: IDLE → PRECHECK → TELEPORT → UNLOAD → DELETE → RECREATE → VERIFY → COMPLETE. Each phase is persisted for recovery.

🗂️

Unified /rwr Command Tree

All admin and player commands are consolidated under a single /rwr root with tab-completion and built-in help via /rwr help.

🔄

Automatic Config Migration

On first boot with a v3 config, the plugin automatically migrates to the v4 schema and creates a timestamped backup at config.v3.backup.yml.

📊

Operator Visibility

/rwr status exposes exact current phase, failure reason, resume state, and schedule. /rwr next shows exact countdown to next reset.

✨Features

Core Functionality

  • ✓ Automated world resets on daily, weekly, or monthly schedules
  • ✓ Selective region resets — reset specific 512×512 block areas without deleting the whole world
  • ✓ GUI-based configuration — no manual YAML editing required
  • ✓ Multiverse-Core integration (MV4 API; MV5 best-effort import)
  • ✓ Async world deletion and recreation to avoid main-thread blocking
  • ✓ Region batch chunk unloading (16 chunks/tick) to prevent lag spikes

Safety & Reliability

  • ✓ Phase-aware reset state machine — exact tracking of where a reset is in the pipeline
  • ✓ Reset state persistence — current phase saved to disk; survives restarts
  • ✓ Automatic retry on failure — up to 3 attempts with exponential backoff (30s, 60s, 120s)
  • ✓ Safe player teleportation — searches up to 512-block radius for solid ground before teleporting
  • ✓ Incomplete reset detection on startup with admin alerts and auto-resume after 60 seconds
  • ✓ PreResetEvent is cancellable — other plugins can abort a reset
  • ⚠ Preflight gates (TPS/players/disk) — config keys included, full enforcement targeted for v4.1.0

Player Experience

  • ✓ World teleport GUI — /rwr tp opens paginated world selection menu
  • ✓ /rwr back — return to your previous location; smart redirect if resource world was reset
  • ✓ Configurable countdown warnings broadcast to all players before reset
  • ✓ Safe respawn and teleport event interception — guarantees solid ground landing

📦Installation

  1. Download the latest ResourceWorldResetter.jar from the GitHub Releases page.
  2. Back up your server: at minimum, back up the plugins/ResourceWorldResetter/ folder and any worlds you plan to designate as resource worlds.
  3. Install Multiverse-Core 4.3.1+ if not already present. Place its jar in plugins/.
  4. Place ResourceWorldResetter.jar into your server's plugins/ directory.
  5. Start the server. On first boot, the plugin generates a default config.yml with v4 schema. If upgrading from v3, config is automatically migrated and backed up.
  6. Create a resource world if you don't have one already:
    /mv create Resources normal
  7. Assign the resource world using the GUI:
    /rwr gui → Click "Change World"
  8. Verify setup with status commands and run a test reset during a low-activity window.
✅
Confirming a successful install

Run /rwr status. You should see phase: IDLE, the configured world name, and the next scheduled reset time. Run /rwr next to see the countdown.

🚀First Setup

After installation, follow these steps to configure your first resource world reset cycle.

Step 1 — Select Your Resource World

Open the admin GUI with /rwr gui and click Change World. A paginated list of all loaded worlds appears. Click the world you want to designate as the resource world. Alternatively, set it directly in config.yml:

worldName: "Resources"

Step 2 — Choose a Reset Schedule

Click Reset Type in the GUI to select daily, weekly, or monthly. Then set the hour using Restart Time (0–23 in 24-hour format). For weekly or monthly modes, also select the day of the week or month.

Step 3 — Configure Warning Time

Click Warning Time to set how many minutes before the reset players receive a broadcast warning. Common choices: 5 or 10 minutes. Set to 0 to disable warnings.

Step 4 — Test the Reset

During a low-activity window, force a manual reset to verify everything works:

/rwr reset now

Watch /rwr status update through each phase. After completion, the world should be regenerated and accessible to players.

Step 5 — Verify the Schedule

/rwr next
# Output example:
[RWR] Next reset: 2025-07-15 03:00:00 UTC+8
[RWR] Warning time: 2025-07-15 02:55:00 UTC+8
[RWR] Time remaining: 14h 32m

🗃️Configuration

The configuration file is located at plugins/ResourceWorldResetter/config.yml. It uses schema version 4, which is auto-generated on first run and auto-migrated from v3 on upgrade.

YAML# Config schema version — do not edit
configVersion: 4
 
# Target world to reset. Leave empty — use /rwr gui to select.
worldName: ""
 
# Reset schedule
schedule:
  # "daily" | "weekly" | "monthly"
  mode: "daily"
 
  time:
    hour: 3             # 24-hour format: 0–23
 
  warningMinutes: 5    # Broadcast warning N minutes before reset (0 = no warning)
 
  # For weekly: 1=Monday … 7=Sunday
  # For monthly: 1–31 (clamped to month length)
  day: 1
 
# Preflight safety gates (enforcement planned for v4.1.0)
preflight:
  tps:
    min: 15.0            # Abort if TPS < min (0 = disabled)
    onFail: "delay"     # "delay" | "abort" | "force"
  players:
    max: 0               # Abort if online players > max (0 = disabled)
    onFail: "delay"
  disk:
    freeGB: 5.0          # Abort if free disk < freeGB (0 = disabled)
    onFail: "delay"
 
# Region-based resets
regions:
  enabled: false
  immediateRegenerationOnAdd: true
  list: []               # Format: "regionX,regionZ"
 
# Optional anonymous usage analytics
metrics:
  enabled: false

ℹ️
Permission check model

The root /rwr command has no permission gate — Spigot permits all users to send it. Permission checks are enforced per-subcommand in code. This allows non-admin players to use tp and back while admin operations remain restricted.

🔄Reset State Machine

Every full world reset flows through an explicit sequence of phases. The current phase is tracked in memory and persisted to reset-state.yml after every transition, ensuring recovery is possible even if the server stops mid-reset.

PRECHECK
→
TELEPORT
→
UNLOAD
→
DELETE

→
VERIFY
→
COMPLETE
Any active phase on failure →

Phase Descriptions

PhaseWhat HappensFailure Cause
IDLENo reset in progress. Awaiting scheduled time or manual trigger.—
PRECHECKFires PreResetEvent. Other plugins may cancel the reset here. Validates world existence.Event cancelled by another plugin
TELEPORTAll players in the resource world are teleported to a safe spawn in another world.No alternate world available
UNLOADWorld is unloaded from memory via Multiverse-Core or Bukkit.World locked by another process
DELETEWorld folder is deleted asynchronously. Warns if deletion exceeds 60 seconds.File permission error, disk I/O failure
RECREATEWorld is regenerated using MV or Bukkit WorldCreator. Safe spawn is configured.Multiverse fails to create world
VERIFYConfirms the world is loaded, accessible, and spawn is set.World not loaded after recreation
COMPLETEFires PostResetEvent(success=true). Broadcasts completion. Reschedules next reset.—
FAILEDReset failed at a phase. Failure reason and phase are persisted. Retry scheduled automatically.Up to 3 retries; escalates to admin alert

Failure & Retry Behaviour

When a phase fails, the plugin automatically retries up to 3 times with increasing delays: 30 seconds, 60 seconds, and 120 seconds. After all retries are exhausted, admins receive an in-game alert and the failure is logged. The reset state is preserved so admins can inspect it via /rwr status and manually intervene with /rwr resume.

Restart Recovery

If the server stops while a reset is in progress, the reset-state.yml file is written with the last known phase. On the next startup, the plugin detects this file, alerts online admins, and schedules an automatic resume after 60 seconds. Admins can run /rwr resume to trigger it immediately, or /rwr resume cancel to abort and clear the state.

🗺️Region Resets

Region mode allows you to reset specific sections of the resource world without deleting the entire thing. A region in Minecraft is a 32×32 chunk (512×512 block) area stored in a single .mca file.

Enabling Region Mode

/rwr region enable    # Enable selective region resets
/rwr region disable   # Disable — full world reset instead

Adding Regions

# By region coordinates
/rwr region add 0 0      # Region at (0,0)
/rwr region add -1 2     # Region at (-1,2)
 
# By standing in the region
/rwr region addhere      # Must be in the resource world
 
# Region coordinate from block coordinate:
# regionX = floor(blockX / 512)
# regionZ = floor(blockZ / 512)

Region Reset Process

  1. Players standing within affected regions are teleported to the world spawn.
  2. Regions are processed one at a time from the configured list.
  3. For each region: 1,024 chunks are unloaded in batches of 16 per tick to avoid lag.
  4. Region files (.mca) plus POI and entities sub-files are deleted asynchronously.
  5. RegionPreResetEvent and RegionPostResetEvent fire per region.
  6. Deleted chunks regenerate naturally when players or chunk loaders access them.
⚠️
Immediate regeneration on add

By default (immediateRegenerationOnAdd: true), adding a region via command immediately resets it if the resource world is loaded. Set to false to defer resets to the scheduled time only.

🧭Teleport System

The teleportation system provides safe player movement before resets and player-initiated world navigation.

Safe Location Algorithm

Before teleporting any player, the plugin searches for a safe standing location using the following algorithm:

  1. Start from the target world's spawn location.
  2. Search outward in expanding squares up to 256 blocks, scanning from highest block downward at each position.
  3. A safe spot requires: feet block = air, head block = air, floor block = solid non-liquid.
  4. If no spot found within 256 blocks, fall back to a wider 512-block search in 32-block increments.
  5. If still not found, return spawn coordinates at minimum valid Y (safe fallback).

/rwr back — Smart Return

When a player uses /rwr back, the plugin checks whether the resource world was reset since the previous location was recorded. If yes, it attempts to find a nearby village in the resource world and teleport there. If no village is found, the player is sent to the world spawn instead. The player receives a message explaining the redirect.

Persistent Location Tracking

The plugin intercepts PlayerTeleportEvent, PlayerRespawnEvent, and PlayerMoveEvent to continuously update the last known safe standing location for each player per world. This location is used as the preferred target when the plugin needs to safely move a player.

Java@EventHandler

    if (bossIsActive) {
        event.setCancelled(true);
        Bukkit.broadcastMessage("Reset delayed — boss fight in progress!");
    }
}

🛠️Reset Recovery

Handling an Interrupted Reset

  1. Run /rwr status immediately. Note the Current phase and Failed phase fields.
  2. Determine what the phase implies: TELEPORT = players may be displaced; DELETE = world folder may be partially or fully deleted; RECREATE = world may not be regenerated yet.
  3. Address the root cause (disk space, MV error, permission issue) before resuming.
  4. Run /rwr resume to continue from the last persisted safe phase.
  5. If you want to abandon the recovery entirely: /rwr resume cancel then /rwr reset now.

Phase-Specific Recovery Guide

Stuck PhaseCheckResolution
TELEPORTAre all players out of resource world?Manually teleport remaining players, then /rwr resume
UNLOADIs the world locked by another plugin?Identify conflicting plugin, unload world manually, then resume
DELETEDisk permissions? Disk full?Fix permissions or free space, then resume — partial deletion is safe
RECREATEIs MV healthy? Check MV logs./mv create [worldName] normal manually, then /rwr reload
VERIFYIs world loaded in Bukkit?Load world manually, then resume

🔍Troubleshooting

"Command not found" for old commands

Cause: v4 intentionally removed /rwrgui, /resetworld, /rwrregion, /rwrresume, and /reloadrwr.
Fix: Update all scripts to use the new /rwr subcommands. See the command mapping table in the Migration section.

Migration backup missing after startup

Cause: Config migration did not complete — typically a disk permission error or the data folder could not be created.
Fix: Stop the server, check startup logs for the first migration error, verify folder permissions, and restart.

Unexpected schedule behavior

Cause: Old flat v3 config keys or invalid values carried forward.
Fix: Open config.yml and verify schedule.mode is one of daily / weekly / monthly, schedule.time.hour is 0–23, and schedule.day is appropriate for the mode. Run /rwr reload.

"worldName not found" error

Cause: The worldName in config does not match any loaded world.
Fix: Run /mv list to see loaded worlds. Either create the missing world with /mv create [name] normal or update worldName in config and run /rwr reload.

Reset hangs at DELETE phase

Cause: Large world folder, slow disk, or insufficient permissions.
Fix: Check free disk space. Ensure the server process has write permission on the world container. Consider reducing world border size. Deletion logs a warning if it exceeds 60 seconds.

Players teleported to wrong world on back

Cause: Previous location was in the resource world that has since been reset.
Behaviour: This is intentional. The plugin detects the reset and redirects the player to a nearby village or world spawn instead of invalid coordinates.

Plugin fails to load with config errors

Check startup logs for the specific validation error message. Use yamllint.com to validate YAML syntax. Ensure indentation uses 2 spaces (no tabs), all required keys exist (configVersion, worldName, schedule), and values are in the correct format.

Upgrade Procedure

  1. Stop the server.
  2. Replace the v3 jar with ResourceWorldResetter-4.0.0.jar in plugins/.
  3. Start the server. Watch logs — the migration runs automatically on first boot.
  4. Confirm plugins/ResourceWorldResetter/config.v3.backup.yml was created.
  5. Open config.yml and verify configVersion: 4 and all schedule.* keys are present.
  6. Update all scripts and permission configs from old command names to new /rwr subcommands.
  7. Run in-game: /rwr status, /rwr next, /rwr gui.
  8. Test during low activity: /rwr reset now.
  9. Keep the v3 backup until at least two successful scheduled resets have completed.

Post-Upgrade Verification

  • ✓ config.v3.backup.yml exists
  • ✓ config.yml has configVersion: 4
  • ✓ Old commands return "Unknown subcommand" (expected)
  • ✓ /rwr status shows correct world, schedule, and phase
  • ✓ /rwr next shows next scheduled reset
  • ✓ /rwr region list returns expected entries (if using regions)
  • ✓ No migration errors or command registration errors in startup log

⬇️Rollback to v3

⛔
Stop the server immediately

If a reset is in progress when you decide to roll back, stop the server before any further state changes occur. Attempting a rollback while a reset is running may leave the world in an inconsistent state.

  1. Stop the server.
  2. Restore the v3 jar from your backup to plugins/.
  3. Rename config.yml → config.v4.yml (keep it as a reference).
  4. Rename config.v3.backup.yml → config.yml.
  5. Delete reset-state.yml created by v4. Do not restore any v3 reset-state format — let v3 start fresh.
  6. Start the server.
  7. Verify old commands (/rwrgui, /resetworld, etc.) are available and the schedule looks correct.
⚠️
If rollback also fails

Restore the entire plugins/ResourceWorldResetter/ folder from your pre-upgrade backup. Open a GitHub issue with your startup logs from the first failing boot, your redacted config.yml, and your previous plugin version.
