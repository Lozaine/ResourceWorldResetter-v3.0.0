v4.0.0 — Early Adopter Release

ResourceWorldResetter
Documentation

A Spigot plugin for automated resource-world resets. Features a phase-aware state machine, unified /rwr command tree, safe player teleportation, selective region resets, and bStats analytics — all without server restarts.

☕ Java 25+ required
🔌 Spigot API 26.1+
🌍 Multiverse-Core 4.3.1+ (soft depend)
⚡ MIT License

📖Introduction

ResourceWorldResetter automates deletion and regeneration of a designated resource world on your Minecraft server. Players mine, build, and exhaust resources freely — the world resets on a configurable schedule (daily, weekly, or monthly) without a full server restart.

Version 4.0.0 is a major breaking-change release introducing a unified /rwr command tree, an explicit phase-based reset state machine, versioned configuration schema (configVersion: 4), and enhanced operator visibility.

⚠️
Breaking Change Release

All legacy commands (/rwrgui, /resetworld, /rwrregion, /rwrresume, /reloadrwr) are no longer registered. Update all scripts, cron jobs, and automation to use the new /rwr subcommand tree before upgrading.

What's New in v4.0.0

🔁

Phase-Based State Machine

Resets progress through 8 explicit phases: IDLE → PRECHECK → TELEPORT → UNLOAD → DELETE → RECREATE → VERIFY → COMPLETE. Each phase is persisted to reset-state.yml for restart recovery.

🗂️

Unified /rwr Command Tree

All admin and player commands under a single /rwr root with tab-completion and a built-in help page via /rwr help.

🔄

Automatic Config Migration

On first boot with a v3 config, the plugin automatically migrates to the v4 schema and writes a timestamped backup at config.v3.backup.yml. No manual YAML editing required.

📊

Operator Visibility

/rwr status exposes exact current phase, failure reason, resume state, and schedule. /rwr next shows a precise countdown including server timezone.

🔁

Automatic Retry with Backoff

Failures trigger up to 3 automatic retry attempts at 30 s, 60 s, and 120 s delays. After all retries are exhausted, admins receive an in-game CRITICAL alert.

📈

bStats Analytics

Optional anonymous metrics via bStats (plugin ID 25197). Charts include reset type, hour, warning time, server version, and hours until next reset.

📦Installation

  1. Download RWR-4.0.0.jar from GitHub Releases, Modrinth, or SpigotMC.
  2. Back up your server: at minimum plugins/ResourceWorldResetter/ and any worlds you plan to designate as resource worlds.
  3. Install Multiverse-Core 4.3.1+ (optional but recommended). Place its jar in plugins/.
  4. Place RWR-4.0.0.jar in your server's plugins/ directory.
  5. Start the server. On first boot the plugin generates a default config.yml with configVersion: 4. If upgrading from v3, config is automatically migrated and backed up as config.v3.backup.yml.
  6. Create a resource world if you don't have one:
    /mv create Resources normal
  7. Assign the world using the GUI:
    /rwr gui → Click "Change World"
  8. Verify setup and test during a low-activity window:
    /rwr status   # should show phase: IDLE
/rwr next     # shows next scheduled reset
/rwr reset now # manual test reset
✅
Confirming a successful install

Run /rwr status. You should see phase: IDLE, the configured world name, and the next scheduled reset time. Run /rwr next to see the full countdown with timezone.

🚀First Setup

Step 1 — Select Your Resource World

Open the admin GUI with /rwr gui and click Change World. A paginated list of all loaded worlds appears. Click your resource world. The GUI only shows worlds managed by Multiverse-Core when MV4 is present; otherwise all Bukkit worlds are listed. Alternatively edit config.yml directly:

worldName: "Resources"

Step 2 — Choose a Reset Schedule

Click Reset Type in the GUI to select daily, weekly, or monthly. Click Restart Time to pick the hour (0–23 in 24-hour format). For weekly mode a day-of-week selector opens; for monthly mode a day-of-month selector (1–31) opens.

Step 3 — Configure Warning Time

Click Warning Time to choose how many minutes before the reset players receive a broadcast warning. Options: 0, 1, 5, 10, 15, or 30 minutes. Setting 0 disables warnings.

Step 4 — Test the Reset

During a low-activity window, force a manual reset to verify end-to-end flow:

/rwr reset now

Watch /rwr status update through each phase. After completion the world regenerates and becomes accessible to players.

Step 5 — Verify the Schedule

/rwr next

🗃️Configuration

The configuration file lives at plugins/ResourceWorldResetter/config.yml. It uses schema version 4, auto-generated on first run and auto-migrated from v3 on upgrade. After editing the file manually, run /rwr reload to apply changes.

# Reset schedule

  mode: "daily"

  time:
    hour: 3              # 0–23 (24-hour format); clamped if out of range

  # For monthly: 1–31 (clamped to actual month length)

  enabled: false

Restart Recovery

On startup, the plugin checks for a reset-state.yml file. If found, it reads the persisted phase and resume phase, alerts online admins, and schedules an auto-resume after 60 seconds. Admins can run /rwr resume to trigger immediately or /rwr resume cancel to abort and clear state.

🔃Failure & Retry Behaviour

When any active phase fails, the plugin automatically retries up to 3 times using exponential backoff before escalating to admins. All failures are logged to logs/resource-reset.log with stack traces.

A CRITICAL broadcast is sent server-wide. Online admins receive a direct message with the last failure reason. The next scheduled reset is re-queued normally. The failed state is visible via /rwr status.

Resume Phase Resolution

When resuming an incomplete reset, the plugin uses ResetPhase.resolveResumePhase() with three-tier priority: (1) saved active resumePhase from state file, (2) active failedPhase if current is FAILED, (3) current phase if active — otherwise IDLE. This prevents re-running already-completed phases.

🗺️Region Resets

Region mode resets specific sections of the resource world without deleting the entire thing. A Minecraft region is a 32×32 chunk (512×512 block) area stored in a single .mca file.

Enabling Region Mode

/rwr region enable     # Enable selective region resets
/rwr region disable    # Revert to full world reset
/rwr region list       # Show configured region coordinates

Adding Regions

# By region coordinates (not block or chunk coordinates)
/rwr region add 0 0       # Region at (0,0)
/rwr region add -1 2      # Region at (-1,2)

# By standing in the region (must be inside the resource world)
/rwr region addhere

# Convert block coordinates to region coordinates:
#   regionX = floor(blockX / 512)   (or blockX >> 9)
#   regionZ = floor(blockZ / 512)

Region Reset Process

  1. Players standing within affected regions are teleported to the world spawn of an alternate world.
  2. Regions are processed one at a time from the configured list. RegionPreResetEvent fires per region (cancellable).
  3. For each region: all 1,024 chunks (32×32) are unloaded in batches of 16 chunks per tick to avoid lag spikes.
  4. The .mca, POI, and entities sub-files are deleted asynchronously via CompletableFuture.runAsync().
  5. RegionPostResetEvent fires per region after deletion completes (success or failure).
  6. Deleted chunks regenerate naturally when players or chunk loaders next access them.
⚠️
Immediate regeneration on add

When immediateRegenerationOnAdd: true (default), running /rwr region add immediately resets that region if the resource world is loaded. Set to false to defer resets to the scheduled time only.

🧭Teleport System

The teleportation system provides safe player movement before resets and player-initiated world navigation. It hooks PlayerTeleportEvent, PlayerRespawnEvent, and PlayerMoveEvent to maintain per-player, per-world safe location memory.

Safe Location Algorithm

Before teleporting any player the plugin finds a safe landing spot using a two-pass expanding search:

  1. Start from the target world's spawn. Search outward in expanding squares up to 256 blocks, scanning from highest block Y downward at each XZ position.
  2. A valid spot requires: feet block = air, head block = air, floor block = solid non-liquid.
  3. If nothing found within 256 blocks, a wider 512-block search in 32-block increments is attempted.
  4. Final fallback: return spawn coordinates at minimum valid Y.

/rwr back — Smart Return with Reset Detection

When a player runs /rwr back, the plugin checks whether the resource world was reset after the previous location was recorded (comparing lastResourceWorldResetCompletedAtMillis vs the location's recorded timestamp). If the world was reset:

  1. Attempt to find the nearest village via World#locateNearestStructure (Paper/Spigot API; silently skipped if unavailable).
  2. If a village is found: teleport there and notify the player with a redirect message.
  3. If no village found: teleport to the resource world spawn.

Post-Respawn Safety

After intercepting a PlayerRespawnEvent, a one-tick delayed task resets the player's fall distance and grants 40 ticks of no-damage time to prevent immediate fall damage at the safe respawn location.

TeleportGUI World Filter

The /rwr tp GUI shows only worlds with Environment.NORMAL. Nether and End worlds are filtered out to prevent accidental teleportation to hostile environments without context.

📡Custom Events

ResourceWorldResetter exposes four Bukkit events for integration with other plugins. All fire on the main thread.

Event ClassCancellableWhen FiredKey Fields
PreResetEvent ✅ Yes PRECHECK phase — immediately before a full world reset begins worldName, resetType
PostResetEvent ❌ No COMPLETE phase or after final retry failure worldName, resetType, success
RegionPreResetEvent ✅ Yes Before each individual region is reset; cancelling skips that region worldName, regionX, regionZ
RegionPostResetEvent ❌ No After each individual region reset attempt worldName, regionX, regionZ, success

Example: Cancelling a Reset During a Boss Fight

Java@EventHandler
public void onPreReset(PreResetEvent event) {
    if (bossArenaManager.isBossActive()) {
        event.setCancelled(true);
        Bukkit.broadcastMessage("[MyPlugin] Reset delayed — boss fight in progress!");
    }
}

Example: Logging Region Resets

Java@EventHandler
public void onRegionPost(RegionPostResetEvent event) {
    getLogger().info("Region (" + event.getRegionX() + "," + event.getRegionZ()
        + ") reset " + (event.isSuccess() ? "successfully" : "with errors"));
}

🛠️Reset Recovery

Handling an Interrupted Reset

  1. Run /rwr status. Note Current phase, Failed phase, Resume phase, and Last failure.
  2. Understand the phase: TELEPORT = players may be displaced. DELETE = world folder may be partially removed. RECREATE = world may not exist yet.
  3. Address the root cause (disk space, MV error, permission issue) before resuming.
  4. Run /rwr resume to continue from the last persisted safe phase boundary.
  5. To abandon recovery entirely: /rwr resume cancel then /rwr reset now.

Phase-Specific Recovery Guide

TELEPORTAre all players out of resource world?Manually teleport remaining players; then /rwr resume

DELETEDisk permissions? Disk full? Check log for "deletion took Xms exceeding 60000ms"Fix permissions or free space; partial deletion is safe to resume from
RECREATEIs MV healthy? Check MV startup logs./mv create [worldName] normal manually, then /rwr reload

🔍Troubleshooting

"Command not found" for old commands

Cause: v4 intentionally removed /rwrgui, /resetworld, /rwrregion, /rwrresume, and /reloadrwr from plugin.yml.
Fix: Update all scripts and macros to the new /rwr subcommands. See the command mapping table in the Migration section.

Migration backup missing after startup

Cause: Config migration failed to complete — typically a disk permission error or the data folder could not be created.
Fix: Stop the server, check startup logs for the first migration error, verify folder write permissions, and restart.

Unexpected schedule behaviour after migration

Cause: Old flat v3 config keys or invalid values were carried forward.
Fix: Open config.yml and verify: schedule.mode is daily, weekly, or monthly (lowercase); schedule.time.hour is 0–23; schedule.day is 1–7 (weekly) or 1–31 (monthly). Run /rwr reload.

"worldName not found" error

Cause: The worldName in config does not match any loaded world.
Fix: Run /mv list to see loaded worlds. Create the missing world with /mv create [name] normal or update worldName in config and run /rwr reload.

Reset hangs at DELETE phase

Cause: Large world folder, slow disk, or insufficient file permissions. The log will show "deletion took Xms exceeding 60000ms" as a WARNING.
Fix: Check free disk space. Ensure the server process has write permission on the world container. Consider reducing world border size.

Plugin fails to load with config errors

Check startup logs for the specific validation error. Use yamllint.com to validate YAML syntax. Required keys: configVersion, worldName, and the schedule block. Indentation must use 2 spaces — no tabs.

Players teleported to wrong world on /rwr back

Cause: Previous location was in the resource world that was reset since the location was recorded.
Behaviour: Intentional. The plugin compares the location recorded-at timestamp against lastResourceWorldResetCompletedAtMillis and redirects to a nearby village or world spawn. The player receives an explanatory message.

TPS detection always returns 20.0

Cause: Running on vanilla Spigot without Paper; NMS reflection for recentTps failed.
Fix: Switch to Paper for accurate TPS reporting. On standard Spigot, TPS defaults to 20.0 (logged as a WARNING).

  • ✓ No migration errors or command registration errors in startup log
  • ✓ Successful /rwr reset now test completed

⬇️Rollback to v3

⛔
Stop the server immediately if a reset is in progress

Attempting a rollback while a reset is active may leave the world in an inconsistent state. Stop the server before proceeding.

  1. Stop the server.
  2. Restore the v3 jar from your backup to plugins/.
  3. Rename config.yml → config.v4.yml (keep as reference).
  4. Rename config.v3.backup.yml → config.yml.
  5. Delete reset-state.yml created by v4. Do not restore any v3 reset-state format — let v3 start fresh.
  6. Start the server.
  7. Verify old commands (/rwrgui, /resetworld, etc.) are available and the schedule is correct.
⚠️
If rollback also fails

Restore the entire plugins/ResourceWorldResetter/ folder from your pre-upgrade backup. Open a GitHub issue with startup logs from the first failing boot, your redacted config.yml, and your previous plugin version.

ResetPhase Enum

Javapublic enum ResetPhase {
    IDLE, PRECHECK, TELEPORT, UNLOAD,
    DELETE, RECREATE, VERIFY, COMPLETE, FAILED;

    // true for COMPLETE or FAILED
    public boolean isTerminal();

    // true for PRECHECK through VERIFY
    public boolean isActive();

    // Returns next phase in sequence; terminal phases return themselves
    public ResetPhase next();

    // Null-safe, case-insensitive; returns IDLE for unknown/blank input
    public static ResetPhase fromString(String value);

    // Priority: savedResumePhase (if active) → failedPhase (if FAILED) → currentPhase
    public static ResetPhase resolveResumePhase(
            ResetPhase currentPhase, ResetPhase failedPhase, ResetPhase savedResumePhase);
}
